# 개인 활동 허브 (Personal Brand Hub) 구축 프로젝트 - 1단계: 요구사항 분석서 (Requirements Analysis)

## 1. 프로젝트 개요 및 배경

### 1.1 배경 및 목적
본 프로젝트는 여러 소셜 채널과 플랫폼에 분산된 개인 활동 데이터를 하나의 정돈된 웹 사이트로 통합하여 퍼스널 브랜드를 강화하는 **"공식 퍼스널 브랜드 허브(Personal Brand Hub)"** 구축을 목표로 합니다. 

이 시스템은 로컬 컴퓨터의 가동 여부에 의존하지 않고 24시간 독립적으로 작동하는 **외부 서비스 지향형 아키텍처**를 채택합니다. 따라서 모든 수집, LLM 처리, 웹 서비스 가동은 **"외부 클라우드(VPS) 서버에 집중하여 가동"**됩니다.

### 1.2 핵심 비즈니스 가치
1. **완전한 클라우드 독립성**: 로컬 PC가 꺼져 있더라도 VPS 상에서 크롤러와 LLM 에이전트가 작동하여 홈페이지 피드가 실시간 최신화됩니다.
2. **동기화 효율성**: 옵시디언 파일 저장소(Vault) 자체를 VPS 클라우드에 호스팅하고 실시간 동기화 서버를 가동하여, 사용자는 모바일/노트북 등 기기 제약 없이 메모를 작성하고 홈페이지에 발행할 수 있습니다.
3. **보안 및 규정 준수**: 우회 세션 쿠키 대신 **공식 LLM API(GPT-4o-mini 또는 Claude 3.5 Sonnet)**를 연동하여, 비용(월 수십~수백 원 수준)을 극소화하면서도 보안 위협 및 계정 정지(Ban) 우려를 근본적으로 차단합니다.

## 2. 상세 요구사항 명세

### 2.1 VPS 중심의 콘텐츠 동기화 요구사항 (VPS-Centric Scraper & Syncer)
- **VPS 상의 옵시디언(Obsidian) Vault 호스팅**:
  - 마크다운 문서 저장소인 옵시디언 Vault 자체를 VPS 내부에 호스팅합니다.
  - 외부 기기(사용자의 모바일, 노트북 등)에서 이 저장소에 실시간 양방향 접근할 수 있도록 **CouchDB 기반의 Obsidian Self-hosted LiveSync 서버**를 VPS 내부의 Docker 컨테이너로 구동합니다.
- **네이버 블로그 크롤러의 VPS 이식**:
  - 네이버 블로그 수집 파이썬 스크립트를 VPS의 백그라운드 프로세스로 가동합니다.
  - 스크립트는 주기적으로 네이버 블로그를 크롤링하여 본문을 마크다운 파일로 변환한 뒤, VPS에 있는 옵시디언 Vault 디렉토리에 직접 저장합니다.
- **소셜 미디어 동기화**:
  - GitHub API 및 LinkedIn 크롤러 등 모든 외부 수집 에이전트가 VPS 서버 내에서 백그라운드 크론탭(Cron) 스케줄러에 의해 상시 구동됩니다.

### 2.2 LLM 연동 요구사항 (공식 API 연동)
- **공식 API 통신 규격**:
  - OpenAI 및 Anthropic의 공식 API 엔드포인트를 호출하여 텍스트 정제 및 요약 작업을 수행합니다.
  - 사용자는 발급받은 API Key를 웹서버 환경 변수(`.env`)에 안전하게 저장합니다.
- **비용 최적화 모델 활용**:
  - 기본 요약 및 태그 분류 등 텍스트 처리는 속도가 빠르고 비용이 극도로 저렴한 **GPT-4o-mini** 모델을 우선 적용합니다. (월 평균 발생 비용 500원 이하 예상)
  - 고품질 비즈니스 인텔리전스 분석이 필요한 영역에 한해 **Claude 3.5 Sonnet** API를 선택적 연동합니다.

### 2.3 백오피스 및 관리 기능 요구사항
- **어드민 웹 UI**: 웹 브라우저를 통해 VPS의 관리 페이지에 접속하여 모든 설정을 제어할 수 있어야 합니다.
- **API Key 관리 및 상태 패널**: 등록된 API Key의 유효 상태 및 잔여 크레딧 경고 등을 백오피스 대시보드에 연동합니다.
- **에이전트 제어판**: 크롤링 주기 설정, 에이전트 동작 로그 확인, 수동 즉시 수집 트리거 버튼을 제공합니다.

### 2.4 VPS 호스팅 사양 및 OS 최적화 요구사항
- **메모리(RAM) 부족 방지**: VPS 단일 서버에서 Next.js 웹서버, CouchDB 동기화 서버, SQLite DB, 그리고 에이전트 스크립트를 함께 가동해야 합니다.
- **Swap 공간 활성화**: OS 단에서 최소 **2GB~4GB 크기의 가상 스왑 메모리(Swap File)**를 필수적으로 설정하여 가동 중 메모리 부족(OOM)으로 인한 전체 서버 다운을 방지합니다.

## 3. 유스케이스 및 데이터 흐름 시나리오

### 3.1 외부 발행 및 자동 렌더링 시나리오
1. **블로그 포스팅**: 사용자가 네이버 블로그에 포스팅을 게시합니다.
2. **VPS 수집**: VPS에서 돌고 있는 파이썬 크롤러가 새 글을 감지하여 VPS 내의 옵시디언 Vault 디렉토리에 마크다운 파일로 저장합니다.
3. **사용자 확인**: 사용자는 이동 중 모바일이나 태블릿의 옵시디언 앱(VPS의 CouchDB와 자동 동기화됨)을 켜서 수집된 마크다운 메모를 확인하고 편집합니다.
4. **발행 요청**: 마크다운 파일 상단(Frontmatter)에 `publish: true` 태그를 기입하고 저장합니다.
5. **LLM 가공**: VPS의 감시 에이전트가 파일 수정을 감지하고, 공식 LLM API를 호출하여 핵심 요약과 태그 분류를 생성합니다.
6. **DB 적재 및 노출**: 가공 결과를 SQLite 데이터베이스에 저장하고 Next.js 홈페이지 피드에 즉시 노출합니다.

### 3.2 예외 대응 시나리오
- **API 한도 초과 (Rate Limit / Quota Exceeded)**: API 호출 실패 시 에러 코드를 감지하여 디스코드/텔레그램 웹훅으로 즉시 경고 알림을 발송하고, 재시도 대기 큐에 등록합니다.
- **수집 중복**: 이미 데이터베이스에 등록된 `originalUrl`을 체크하여 수집 단계에서 중복 가공을 차단합니다.
