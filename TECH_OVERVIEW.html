1 · 프로젝트 개요
swarm56 = 원종석(Jongseok Won)의 퍼스널 브랜드 허브. 8개 소셜 채널에 흩어진 글·활동을 한 곳(swarm56.com)에 모아 카드 피드로 보여준다.

두 가지 목적:

  • 표면 목적 — 채널별 콘텐츠 집계 홈페이지 + 어드민 백오피스.
  • 실제 연구 목적 — 이종 멀티에이전트 협업 워크플로우 확립·사례연구 (구현=Claude, 검증=Codex, 오케스트레이션=Peter 역할 분리). 논문/책 자료화.

핵심 철학: 옵시디언 볼트를 단일 소스(메인 DB)로 두고 — 소셜 → 클리핑 → 볼트 → 파생캐시 → 홈피의 단방향 데이터 흐름.

2 · 요구사항

기능

  • 8개 채널(네이버블로그·Notion·YouTube·GitHub·LinkedIn·Instagram·Facebook·Swarm) 콘텐츠 카드 피드 + 채널 필터.
  • 카드 → 원문 링크(noopener), 본문 이미지 포함.
  • 백오피스: 로그인, 카드 삭제/복원/편집, 수집 로그(SyncRun), "지금 클리핑"/"강제 갱신" 트리거, 감사 로그.
  • 삭제 의도 영속 — 삭제한 카드는 재수집/재파생에도 부활 X.
  • 일 1회 자동 수집.

비기능

  • 단방향 데이터 흐름(볼트→SQLite→홈피, 역방향 쓰기 금지).
  • 멱등성·재개 가능(클리퍼 중단 후 재실행 안전).
  • 홈피·백오피스는 볼트에 쓰지 않음(읽기 전용).
  • 비밀(토큰·키)은 env로만, 코드/Git 금지.
  • 개인 규모(저비용 1GB VPS) — 무거운 인프라 지양.

4 · 아키텍처

4.1 데이터 흐름 (단방향)

소셜 8채널 │ Phase A: 클리핑 (전문 md + 본문이미지 _assets/*.webp + 링크치환 + LLM발췌) ▼ 옵시디언 볼트 (메인 DB) raw/<채널>/*.md + _assets/*.webp │ Phase B: 파생 (활성 suppression URL 스킵 = 삭제의도 보존) ▼ SQLite 파생캐시 (site-v5.db) FeedCard · SuppressionRecord · AdminAudit · SyncRun │ 읽기 전용 ▼ 홈페이지(Next.js) + 백오피스(/admin)
홈피·백오피스는 SQLite만 읽는다. 볼트 원본은 건드리지 않음. 클리퍼만 볼트에 쓴다.

4.2 삭제 영속성 — "방식 B" (Suppression / Tombstone)

삭제 → FeedCard 행 제거 + SuppressionRecord 기록(restoredAt=null) 재파생 → 활성 suppression URL? ── 예: 스킵(부활 X) / 아니오: 카드 생성 복원 → tombstone 해제(restoredAt=now) + 볼트 md에서 카드 재삽입 [하나의 트랜잭션]

삭제 키 = originalUrl (재파생에도 안정). 삭제/복원/편집 + 감사로그는 한 트랜잭션(원자성).

4.3 배포 토폴로지

AWS Lightsail 54.116.19.34 (Ubuntu, Seoul) Nginx :443 (swarm56.com) ├─ proxy → swarm56-web (npm start :3000, /opt/swarm56/app) → site-v5.db └─ /vault/ alias → /var/lib/swarm56/vault-v5/raw/ swarm56-agent.timer (매일 04:00 UTC) → 클리핑 에이전트 → vault-v5 → site-v5.db
주의: 앱의 .env는 Next(@next/env)가 단독 로드한다. systemd EnvironmentFile 사용 금지 — 이중 로드 시 bcrypt 해시의 $가 깨져 로그인 실패(2026-06-29 핫픽스).
